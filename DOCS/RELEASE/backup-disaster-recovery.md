# Backup and Disaster Recovery

## Purpose

This document defines backup and disaster recovery procedures for CuePoint release artifacts and documentation.

## Scope

- Release artifacts (installers, checksums)
- Documentation (docs, README)
- Appcast/update feeds
- Build configuration

## Backup Policy

### What to Backup

| Asset | Location | Frequency |
| --- | --- | --- |
| Release artifacts | GitHub Releases | Per release |
| Documentation | Repository | Per commit |
| Appcast | gh-pages or CDN | Per release |
| Build config | Repository | Per commit |

### Backup Schedule

- **Release artifacts**: Automatically retained by GitHub Releases
- **Documentation**: Git history provides versioning
- **Weekly**: Verify backup integrity (checksums, accessibility)

## Backup Procedures

### 1. Release Artifacts

- GitHub Releases stores artifacts with each release
- Do not delete releases; mark as "Pre-release" or "Withdrawn" if needed
- Export critical releases to secondary storage (e.g., S3, external drive) for major versions

### 2. Documentation

- All docs in `docs/` are version-controlled
- Tag releases to preserve doc state: `git tag v1.2.3`
- Supersede or update design docs when they change (no separate archive folder)

### 3. Appcast

- Appcast is generated by CI; source of truth is release workflow
- Keep copy of last-known-good appcast in repo or backup location
- Document appcast URL and regeneration steps in [Update Feed Recovery Runbook](update-feed-recovery-runbook.md)

## Disaster Recovery

### Scenario: GitHub Repository Compromised

1. Verify integrity of local clone
2. Create new repository if necessary
3. Push from trusted backup
4. Re-create releases from artifact backups
5. Update appcast URLs if repository moved
6. Notify users via release notes

### Scenario: Release Artifacts Lost

1. Re-build from release tag: `git checkout vX.Y.Z`
2. Run CI/build scripts to regenerate artifacts
3. Re-sign and re-notarize (macOS)
4. Re-publish to GitHub Releases
5. Regenerate appcast

### Scenario: Appcast Corrupted

1. Follow [Update Feed Recovery Runbook](update-feed-recovery-runbook.md)
2. Regenerate appcast from release metadata
3. Re-publish to host
4. Verify update flow

## Backup Verification

### Monthly

- [ ] Verify release artifacts downloadable
- [ ] Verify appcast URLs return valid XML
- [ ] Test restore from tag (build from source)

### Quarterly

- [ ] Full restore drill: build from tag, verify output
- [ ] Review backup retention (keep last 3 major versions)
- [ ] Update this document if procedures change

## Retention

- **Release artifacts**: Retain indefinitely for supported versions
- **Support bundles** (user-submitted): 30 days, then delete
- **Logs** (local): Per log retention policy (see operations docs)
