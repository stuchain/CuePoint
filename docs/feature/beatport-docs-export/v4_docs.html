OAuth2 Authentication

Prerequisites

Never publically share your client secret. JavaScript applications with no server backend can use the browser redirect flow. Server side applications can use the secret based flow. You can use one of the following mechanisms to obtain an access token.

Some of the examples shown here rely on HTTPie, a CLI utility for making http requess.

Step 1: Obtaining an Access Token

Authorization Code Grant Flow

Your OAuth Application will be configured to accept one or more redirect URIs. These URIs should be managed by you and should be configured to accept a code URL parameter which you will exchange for an access token.

# in a browser
https://api.beatport.com/v4/auth/o/authorize/?client_id=<CLIENT_ID>&response_type=code&redirect_uri=<REDIRECT_URI>

Hit "Authorize" to accept the scope grants. A grant token code will be returned to the redirect URI as a url param code, if it is a valid URI for the application.

Exchange grant token code for an access token.

CLIENT_ID="YOUR_CLIENT_ID"
CODE="CODE_FROM_BROWSER_REDIRECT"

# note the `-f` option for form-encoding
http -f https://api.beatport.com/v4/auth/o/token/ \
  client_id=$CLIENT_ID \
  code=$CODE \
  grant_type=authorization_code \
  redirect_uri=https://api.beatport.com/v4/auth/o/post-message

User Password Grant Flow

Your OAuth application should have a client id, client secret, username and password that can be used to obtain an access token.

CLIENT_ID="YOUR_CLIENT_ID"
CLIENT_SECRET="YOUR_CLIENT_SECRET"
USERNAME="VALID BEATPORT USER NAME"
PASSWORD="PASSWORD ASSOCIATED TO USERNAME"


 # note the `-f` option for form-encoding

http -f https://api.beatport.com/v4/auth/o/token/ \
 client_id=$CLIENT_ID \
 client_secret=$CLIENT_SECRET \
 username=$USERNAME \
 password=$PASSWORD \
 grant_type=password

Client Credentials Grant Flow

Your OAuth application should have a client id and secret that can be used to obtain an access token.

CLIENT_ID="YOUR_CLIENT_ID"
CLIENT_SECRET="YOUR_CLIENT_SECRET"

http -f https://api.beatport.com/v4/auth/o/token/ \
    client_id=$CLIENT_ID \
    client_secret=$CLIENT_SECRET \
    grant_type=client_credentials

Access Token Response

After succesfully authorizing using the client or server based flows, you should have received a response that looks something like the following:

{
    "access_token": "YOUR_ACCESS_TOKEN",
    "refresh_token": "YOUR_REFRESH_TOKEN",
    "expires_in": 36000,
    "scope": "...",
    "token_type": "Bearer"
}

Step 2: Using the Access Token

Once you have obtained an access token, you can use it to make authenticated requests by applying it as a header to each request.

For example, introspecting the current authorized user:

ACCESS_TOKEN="YOUR_ACCESS_TOKEN"

http https://api.beatport.com/v4/auth/o/introspect/ \
  "Authorization:Bearer $ACCESS_TOKEN"

Step 3: Refreshing the Access Token

CLIENT_ID="YOUR_CLIENT_ID"
ACCESS_TOKEN="YOUR_ACCESS_TOKEN"
REFRESH_TOKEN="YOUR_REFRESH_TOKEN"

http -f https://api.beatport.com/v4/auth/o/token/ \
  "Authorization:Bearer $ACCESS_TOKEN"
  client_id=$CLIENT_ID \
  refresh_token=$REFRESH_TOKEN \
  grant_type=refresh_token

Step 4: Revoking the Access Token

CLIENT_ID="YOUR_CLIENT_ID"
ACCESS_TOKEN="YOUR_ACCESS_TOKEN"

http -f POST https://api.beatport.com/v4/auth/o/revoke/?client_id=$CLIENT_ID&token=$ACCESS_TOKEN

Error Codes

We aim to have consistent error codes across our API. In general, the following should error codes should be expected. If there is special behavior per endpoint, please see the endpoint specific documentation.

All errors codes should be in JSON format

GET requests

GET item by ID with non-existent ID

If the user perfroms GET on a "get by id" endpoint with an {id} that does not exist, the user should receive a 404 Not Found error with the following response body:

{
  "detail": "Not found."
}

Examples:

  • /v4/catalog/tracks/{id}/

GET item by ID with ID the user does not have access to

If the user perfroms GET on a "get by id" endpoint with an {id} that they do not have access to, the user should receive a 404 Not Found error with the following response body:

{
  "detail": "Not found."
}

Applying filters on GET requests with invalid values

If the user tries to filter results by an invalid data type, the user should receive 400 Bad Request error with a JSON response body that explains the error made.

Expected response body when using a string for id field:

{
      "id": [
          "Enter a number."
      ]
  }

Expected response body when using an invalid value while filtering on a boolean or enum field:

{
    "<query>": [
        "Select a valid choice. <value> is not one of the available choices."
    ]
}

Examples:

  • Entering a string to filter by id i.e.: /v4/catalog/artists/\?id\=kh
  • Entering any string besides true or false on a true/false field i.e. /v4/catalog/artists/?enabled=hfgd

DELETE requests

DELETE on a non-existent id

When performing a DELETE request on an {id} that does not exist, the user should receive a 404 Not Found error with the following response body:

{
  "detail": "Not found."
}

DELETE on a an ID that a user is not permitted to delete

When performing a DELETE request on an {id} that does not exist, the user should receive a 403 Forbidden error with the following response body:

{
    "detail": "You do not have permission to perform this action."
}

PATCH requests

PATCH with invalid data

When performing a PATCH request without all of the required fields, the user should receive a 400 Bad Request error with the following response body:

{
    "name": [
        "This field is required."
    ]
}

PATCH with empty body

When performing a PATCH request without all of the required fields, the user should receive a 400 Bad Request error with the following response body:

{
    "name": [
        "This field is required."
    ]
}

POST requests

POST with invalid data

When performing a POST request without all the required fields, the user should receive a 400 Bad Request error with the following response body:

{
    "track_id": [
        "This field may not be blank."
    ]
}

Incorrect response body:

{
    "track_id": [
        "This field may not be null."
    ]
}
